## Objectif de la séance
La gestion d'erreurs : `try` / `except` (dernière brique des fondamentaux Phase 1).

## Révision éclair
- `and` vs `or` (invalide/dehors) : **`or`**, juste.
- `-u` : **enfin pleinement correct** — « enregistrer la branche vers laquelle on push pour ne plus la renseigner », **sans « fichiers »**. → ACQUIS.
- 3 symboles pièges : puissance = `**` juste ; mais pour « et/ou logiques » il a redit `&` et « V » → **retombé dans le piège**. Recorrigé : la logique = des **MOTS** `and`/`or`/`not`, jamais `&`/`|`/`^`. À redriller (confond encore symbole binaire et mot-clé).

## Ce qui a été vu
- `try` / `except` : tenter un code risqué, attraper l'erreur au lieu de planter.
- **`ValueError`** = l'erreur levée par `int()` sur du texte non numérique.
- Principe : si le `try` réussit, le `except` est ignoré ; s'il échoue, saut direct dans le `except`.

## Exercice fait — `python\test_erreur.py`
```python
nb = input("Indiquez un nombre: ")
try:
    int(nb)
    print("nombre valide: ", nb)
except ValueError:
    print("Ce n'est pas un nombre !")
```
- Fonctionne : `24` → valide, `abc` → message propre, pas de crash. **Structure try/except correcte du 1er coup.**

## Difficultés / observations
- Mécanisme `try`/`except` saisi immédiatement, bonne structure.
- Nuance soulevée : `int(nb)` utilisé comme simple **test** (résultat jeté). Pour réutiliser le nombre → `nb = int(nb)`. Servira à l'intégration.
- Point à driller encore : **mots logiques `and`/`or`/`not` vs symboles binaires `&`/`|`/`^`** (confusion persistante, comme `^` ≠ puissance).
